# CellFlow — Maintainability Review (organization & dead-code pass)

Date: 2026-07-03 · Version reviewed: 0.2.0 · Branch: `claude/codebase-analysis-refactor-ddksg6`

Scope: full source tree (44.8k LOC across 159 modules), with emphasis on
**dead code, test-only surface, and organization/architecture** — deliberately
complementary to the two prior bug-focused passes (`REVIEW.md` 2026-06-23,
`REVIEW-2026-07-02.md`). Nothing already recorded there is repeated. The prior
reviewers verified the load-bearing numerics (MSD, circularity, IoU, order
parameter, divergence sign/axis, ICM cost) as correct; this pass does not
re-litigate them.

Every dead-code / test-only claim below was verified by whole-repo reference
counts (`grep -rnw <symbol> src tests`), accounting for the four dynamic-dispatch
escapes that make automated tools (vulture) unreliable here: napari manifest
entry points (`napari.yaml`), `Quantifier`/`AnalysisPlugin` subclass
auto-registration, Qt override methods, and `.connect()` signal targets.

## Changes applied in this pass

The safe, unambiguous items were executed and verified (full suite: 1261 passed,
44 skipped, 0 failures; `ruff check src tests` clean; net −750 LOC):

1. **Lint** — `ruff --fix` for F401 + RUF100, plus the pre-existing UP045/UP035
   annotations that the newer ruff surfaces; the tree is now ruff-clean. (One
   `# noqa: E402` on `napari/__init__.py:5` is genuine — the import follows a
   function *call* — and was preserved.)
2. **Dead code deleted** — `core/paths.stage_dir` + its now-orphaned `STAGE_DIRS`;
   `correction/labels.cleanup_movie`; `napari/ui_style.tiny_button`; the entire
   `core/label_store` per-frame API (`write_tracked_frame`, `read_tracked_frame`,
   `tracked_n_frames`, `tracked_frame_exists`) **and** the now-orphaned
   `.frames.json` sidecar machinery it existed to serve (`_record_written_frames`,
   `_frames_sidecar`, and the write in `write_full_tracked_stack`) — removing both
   the O(T²) footgun and dead write-only output; `napari/_catalog_table.py`
   (337 lines, 5 classes) + its dedicated test.
3. **`napari/correction/` subpackage** — the 20 `_correction_*` modules + the
   three correction widgets (`correction_widget`, `nucleus_correction_widget`,
   `cell_correction_widget`) moved via `git mv` (history preserved as renames);
   all 47 importing modules across `src/` and `tests/` rewritten to the
   `cellflow.napari.correction.<module>` path. The four controllers
   (`lineage_canvas_controller`, `track_path_controller`,
   `validated_overlay_controller`, `candidate_gallery_controller`) were left in
   place — `track_path_controller` has an external consumer
   (`contact_visualization.py`), so folding them in is a cleaner second step.

**Deliberately NOT executed** (see §2): the scattered test-only functions in
`tracking_ultrack`, `validation_state`, and `core/lineage`. These live in
subpackages the README declares as public scripting API, and some are flagged
"intentional ad-hoc API" by the prior review — they need an intent call, not
blind deletion. One class among them is not dead at all: **test seams** like
`_run_blocking` (present in two widgets) are synchronous entry points that exist
*so that* async worker code is testable. "No production caller" ≠ dead when the
function's job is to be called by tests.

## Summary

The codebase is disciplined and well-tested (136 test files). The maintainability
drag is **not** correctness — it is **surface area**: (1) a body of code that no
production path calls but that tests keep alive (giving false "it's covered, so
it's used" signals), and (2) the `napari/` package, which is 59% of the tree
(26.4k LOC) in 64 flat modules with no internal structure. The single highest-
leverage change is architectural: give the interactive-correction subsystem its
own subpackage.

## 2. Production-dead, test-only (the notable pattern)

These have **zero production callers** — every reference is either the definition
or a test. None is re-exported as public API (not in any `__init__.py`, not in
`docs/`). Tests here pin the behavior of code nothing runs, which reads as
coverage but is really maintenance debt: the module looks live, so it never gets
pruned. Recommend deleting each function **together with its test**, or, if a
function is intended as public scripting API, promoting it to a real
`__init__.py` export so the intent is explicit.

**`_catalog_table.py` — the whole module (337 lines, 5 classes).** `CatalogTable`,
`CatalogRowSpec`, `_CatalogRow`, `_SeparatorRow` have zero production references;
the catalog UI is served entirely by `_experiments_panel.py::ExperimentsPanel`.
The only thing referencing `_catalog_table.py` is `tests/napari/test_catalog_table.py`.
This is a superseded parallel implementation kept alive solely by its own test.
→ Delete `_catalog_table.py` + `test_catalog_table.py`.

**`core/label_store.py` — the entire per-frame API.** `write_tracked_frame`,
`read_tracked_frame`, `tracked_n_frames`, `tracked_frame_exists` have **no
production callers**. Production uses only `write_full_tracked_stack` /
`read_full_tracked_stack`. This is the O(T²) per-frame path that `REVIEW.md` #4
replaced with the batched writer — the old functions were left behind. Note
`tracked_frame_exists` was even *fixed* (sidecar) in `REVIEW.md` #9 for a caller
that no longer exists. Keeping `write_tracked_frame` around actively invites the
quadratic-write mistake to recur. → Delete the per-frame four + their tests.

**`tracking_ultrack/validation_state.py` — frame-level validation writers.**
`write_validated_frames`, `validate_frame`, `invalidate_frame`, `is_validated`,
`read_validated_cells_at_frame`, `validate_track` have no production callers.
Validation in production is track-level (`read_validated_tracks` has 23 callers,
`invalidate_track` 4, `remap_validated_tracks` 6). The frame-level writer API and
`validate_track` appear to be a superseded/parallel path. `read_validated_frames`
still has 2 callers, so the frame *reader* is live — prune the writers, not the
whole file.

**`core/lineage.py`** — `first_frame`, `last_frame`, `has_gap`, `lane_for`: no
production callers (test-only).

**`tracking_ultrack`** (test-only, no production caller): `atoms.extract_atoms_stack`,
`atoms.enum_connected_unions`, `db_build.build_ultrack_database`,
`db_query.query_hierarchy_cut_states`, `db_query.render_hierarchy_cut_state`,
`validation_nodes.inject_validated_nodes`, `ingest.apply_link_function`,
`cellpose_runner.layout_has_time`. Some are documented as intentional ad-hoc API
in the prior review — confirm intent, then either export or delete.

**napari widgets** (test-only, no production caller):
`cellpose_segment_track_widget.segment_channel` and `preview_channel_masks`
(module-level functions superseded by the in-widget streaming path),
`divergence_maps_widget._run_blocking`, `contact_visualization._rasterize_track_image`.

> Rule of thumb going forward: a public function whose only non-definition
> references live under `tests/` is either dead or undeclared API. Make the call
> explicit — export it or delete it. A quick CI guard (grep each `def`/`class`
> for a non-test, non-definition reference) would stop this from re-accreting.

## 3. Organization / architecture (the high-leverage change)

**`napari/` is 26.4k LOC (59% of the codebase) in 64 flat modules + 1 subpackage.**
Inside it sits an unmarked ~25-file, ~8k-LOC interactive-correction subsystem:
the 20 `_correction_*.py` modules (3.3k LOC), the three widgets
(`correction_widget.py` 1363, `nucleus_correction_widget.py` 2169,
`cell_correction_widget.py` 1169), plus `candidate_gallery_controller.py`,
`track_path_controller.py`, `lineage_canvas_controller.py`,
`validated_overlay_controller.py`, `_correction_track_accordion.py`. It is
already highly self-cohesive (the `_correction_*` set is imported almost entirely
by `nucleus_correction_widget.py`; nothing outside the cluster reaches in).

→ **Extract `napari/correction/` as a subpackage.** This is mechanical (moves +
import rewrites, no logic change), instantly documents the boundary, shrinks the
top-level `napari/` namespace from 64 to ~40 files, and makes the correction code
reviewable as a unit. `contact_analysis/` already proves the pattern is welcome
here. Candidate second subpackage: the workflow-stage widgets
(`cellpose_*`, `nucleus_*`, `cell_workflow_*`, `divergence_maps_widget`).

**`nucleus_correction_widget.py` (2169 lines) is the single largest module.** It
is described in its own docstring as a "coordinator" over the `_correction_*`
helpers, yet it still carries a lot of concrete logic. Vulture flags several of
its private methods as unreferenced (`_add_correction_image_layer`,
`_on_load_tracked`, `_commit_reassign_ids`, `_validated_correction_for_frame`,
`_remove_unvalidated_from_layer`, `_add_validated_overlay`,
`_place_validated_overlay_below_spotlight`); these need per-method verification
(some may be Qt/signal targets) but the count suggests real prunable surface.
Worth a dedicated dead-method sweep once the subpackage move is done.

**Nucleus vs Cell correction coordinators share a mixin but diverge by copy.**
Both `NucleusCorrectionWidget` and `CellCorrectionWidget` are
`(CorrectionViewStateMixin, QWidget)` that compose an inner `CorrectionWidget`.
The comments in `cell_correction_widget.py` ("Mirrors NucleusCorrectionWidget…")
signal deliberate parallel structure. After the subpackage extraction, a focused
diff of the two would likely surface a shared coordinator base worth pulling up —
but that is a behavior-touching change and should follow, not precede, the move.
