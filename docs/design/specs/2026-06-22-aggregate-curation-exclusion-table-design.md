# Aggregate Quantification: curation exclusion table + filter

**Status:** decided direction, pre-implementation (open questions listed)
**Date:** 2026-06-22
**Scope:** the backend data model for human QC of aggregated measurements тАФ a
separate curation tidy table of exclusion flags, left-joined onto the measurement
tables to filter, with the `.iris` export receiving the filtered data.
**Companions:** the curation *tool* that authors this table
(`2026-06-22-aggregate-curation-tool-napari-design.md`); supersedes the removal
recorded in `2026-06-21-aggregate-quantification-artifact-contract-design.md` ┬з1.

## Statement of need

Some frames and whole positions are unusable (out of focus, debris, a segmentation
that blew up) and must be kept out of the pooled statistics тАФ but the decision is a
human judgement, made by *looking at the images*, and it must carry a recorded
**reason** for the lab record. Curation was removed in commit 95df159 when the
`.iris` export became a pure function with no baked-in human judgement; the
artifact-contract spec explicitly anticipated its return as *"an upstream flag
column + a filter reduce step."* This spec defines that path.

## What we already have (survives)

- The old `curation.py` (removed in 95df159, recoverable at rev `39d0df2`): a
  `read_curation` / `apply_curation` pair that **left-joined** `excluded` /
  `qc_reason` onto a table and defaulted missing rows to *kept* ("filter, don't
  delete"). The join key was the deterministic row `id`.
- Measurement tables are tidy, index-keyed by `(experiment_id, position_id, frame,
  cell_id)` (cell objects) / `(experiment_id, position_id, cell_id)`. The same keys
  the NLS `class_label` sidecar is joined on.
- `RunConfig` (`config.py`) already parses an optional, relocatable run-config; a
  `curation` key slots in beside `catalog` / `export_dir`.

## Decisions

### 1. Curation is its own tidy table тАФ a separate, git-versioned artifact.

It is **not** baked into the measurement tables. The measurement tables stay the
*disposable, automatic-derived* outputs (regenerated by any run); the curation
table is the *curated-derived* artifact (human judgement, kept in git, never
overwritten by a run). They are combined only at filter time. This preserves the
artifact-contract's curated/derived split.

### 2. Granularity: frame and whole-position only (no cell-level).

A curator excludes a **frame** of a position, or an **entire position**. Cell-level
exclusion is out of scope (YAGNI тАФ not requested, and far heavier UX).

### 3. Schema.

One row per exclusion. Columns:

| column | meaning |
|---|---|
| `experiment_id` | replicate / series identity (matches the measurement tables) |
| `position_id` | position identity |
| `frame` | the excluded frame index; **empty/NA = the whole position** (every frame) |
| `excluded` | bool, always `True` for a present row (column kept explicit for readability + future "un-exclude" overrides) |
| `exclusion_reason` | free-text reason (required) |

Position-level exclusion is the `frame`-is-NA row. (See Open #1 for the alternative
explicit-`scope` representation.)

### 4. Join + filter: left-join, default kept, applied at export.

`apply_curation(table, curation)` returns the table with each row marked excluded
iff it matches a curation entry by **either**:
- `(experiment_id, position_id, frame)` тАФ a frame-level exclusion, or
- `(experiment_id, position_id)` with the curation `frame` NA тАФ a position-level
  exclusion.

Rows with no matching entry default to kept. Nothing is mutated or deleted in the
measurement source. The **export step** (`pipeline.export` / `run`) applies the
filter before writing `.iris`, so the premade SuperPlots see only kept data. The
on-disk tidy measurement CSVs stay pure (all rows); the exclusion table beside them
carries the flags + reasons for the record. The `.iris` is still a pure function of
its inputs тАФ now `(table, curation)` rather than `(table)`.

### 5. Wiring.

- `RunConfig` gains an optional `curation: Path` key (resolved like `catalog`,
  relative to the config dir; default location `curation.csv` beside the config).
- `run()` loads it (when present) and threads it into `export`, which left-joins +
  filters per table before the Iris write.
- A new `curation.py` (restored + reworked from `39d0df2`) holds
  `read_curation` / `apply_curation`, now keyed on the natural keys of Decision 4
  instead of the row `id`.
- Reuses the `.iris` provenance to record that a curation filter was applied (which
  file, how many rows dropped), so a bundle is self-describing.

## Open / deferred

1. **Position-level representation** тАФ `frame`-is-NA (Decision 3) vs. an explicit
   `scope тИИ {frame, position}` column. NA is terser; explicit `scope` is
   self-documenting and avoids "is blank a typo?" ambiguity. Lean NA, decide at
   implementation.
2. **Frame ranges** тАФ the table stores individual frame rows; a range exclusion
   (frames 40тАУ60) is many rows the *tool* expands. No range syntax in the table
   itself. (Confirm acceptable.)
3. **Un-exclude / overrides** тАФ keeping `excluded` as an explicit column leaves room
   for a future `False` override row, but no precedence rule is defined yet. Out of
   scope now.
4. **One file per series vs per project** тАФ assumed one curation CSV per run-config
   (per series). Cross-series curation is not modelled.
